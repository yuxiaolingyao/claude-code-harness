# 全局开发规约（Claude Code Harness 基线）

**本文件适用于所有项目。项目级 CLAUDE.md 可追加技术栈约束，不可覆盖本文件定义的安全规则。**

> 以下为原则（跨模型通用）。Hook 层是模型中立的安全底线——无论后端用 Claude/GPT/DeepSeek/Gemini，防火墙始终生效。
> 多模型适配详情见 `docs/MODEL_NOTES.md`。

---

## 1. 安全驾驭架

### 1.1 禁止假设

- 禁止假设任何 API 方法签名、配置值、包名。不确定时搜索源码或文档。
- 禁止假设业务逻辑边界。必须基于代码证据。
- **不确定就问，禁止猜。**

### 1.2 五阶段强制流

1. **理解需求** → 2. **阅读分析** → 3. **方案设计**（列出改动文件与影响面） → 4. **获授权准入** → 5. **执行验证**

### 1.3 终端门禁

| 命令类型 | 策略 |
|---------|------|
| 只读（ls, grep, git status, git diff, git log, 编译, 读文件） | 允许，告知目的 |
| 写入（rm, git commit, git push, 部署） | 必须获授权 |
| 安装（pip install, npm install, apt） | 必须获授权 |

### 1.4 强制只读锁定

**未获得"授权"/"改"/"实现"/"可以" + 具体操作之前，禁止调用写工具。**

- 用户说"看一下""分析一下""阅读"均为只读。
- 每次代码建议末尾提示"需要授权后执行"。
- 改动完成后列出变更文件清单。

---

## 2. 编码规范（通用基线）

- 禁止魔法值，使用常量或枚举。
- 禁止捕获 RuntimeException / Exception 而不处理。
- 禁止吞异常（空 catch 块）。
- 禁止 System.out 日志，统一使用日志框架。

> 语言特定规范（Java 禁止 lambda、构造器注入等）应在项目级 CLAUDE.md 中定义。

---

## 3. 交互原则

- **精简输出**：用表格和代码引用，不写长段落。
- **诚实标注**：不确定的 API 标注"待验证"。
- **先方案后执行**：改动前列出影响文件，授权后动手。

---

## 4. 运行熔断

- 同一编译错误连续 2 次修改未解决 → 停止，报告给用户。
- 编译通过即任务完成，禁止在未授权下运行项目。

---

## 5. 上下文卫生

- **两错熔断**：同一问题被纠正 2 次 → 停止，报告用户，不自行继续尝试。
- **任务隔离**：不相关任务之间使用 `/clear` 刷新会话。长会话累积导致准确率下降。
- **禁止臆造**：不确定的 API/配置/路径必须在回答中标注"待验证"，禁止猜测后直接使用。
- **禁止过度设计**：不添加任务不需要的抽象、工具函数、未来预留。

## 6. 跨项目适配

- 编译/检查命令按项目 CLAUDE.md 定义的执行（如 maven/gradle/npm），无定义时询问用户。
- 未识别项目类型时，先读取项目文件结构再建议技术方案，不假设技术栈。
